# Проект Dominoes

Этот документ — основной источник актуальных продуктовых и инженерных решений проекта Dominoes. Подробные правила игры вынесены в [GAME_RULES.md](GAME_RULES.md), а эксплуатационные инструкции — в [runbooks](runbooks).

## Статусы возможностей

- **Реализовано** — возможность присутствует в текущем Web MVP и поддерживается кодом проекта.
- **Следующий этап** — согласованная работа после стабилизации текущего Web MVP.
- **Будущее** — принятое направление развития без обязательства по сроку выпуска.

Статус относится к состоянию репозитория, а не к доступности конкретного production-окружения.

## Назначение продукта

Dominoes — многоплатформенное приложение для игры в Draw Dominoes. Пользователь сможет создать единый аккаунт с уникальным публичным username и играть:

- с компьютером;
- с другим человеком по приватной ссылке;
- в браузере на десктопе и мобильном устройстве;
- в дальнейшем — через Telegram, Android и iOS.

Первый продуктовый релиз сосредоточен на адаптивном браузерном приложении. Все клиенты должны использовать единые серверные правила, аккаунты и версионированный API.

## Текущий Web MVP

### Реализовано

- адаптивное PWA на Next.js для десктопных и мобильных браузеров;
- интерфейсы на украинском, английском и русском языках, язык по умолчанию — украинский;
- регистрация и вход через Supabase Auth;
- профиль с уникальным публичным username;
- матч один на один против детерминированного AI средней базовой сложности;
- приватный матч с человеком по одноразовой ссылке;
- гостевой вход в приглашённый матч без предварительной регистрации;
- серверная проверка игровых команд и защита скрытых костей;
- обновления матча через Socket.IO;
- восстановление актуального состояния матча с сервера;
- локальный development principal при запуске без Supabase; в production он отключён;
- автоматические unit, integration и браузерные проверки основных сценариев.

### Следующий этап

- развернуть и проверить production-окружения Vercel, Render и Supabase;
- закончить сквозную проверку регистрации, восстановления сессии и гостевого присоединения в production;
- подключить мониторинг ошибок, метрики, алерты и privacy-safe аналитику;
- проверить резервное копирование и восстановление на реальном production-процессе;
- провести нагрузочное, security- и accessibility-тестирование перед публичным запуском;
- доработать UX на основании тестирования на реальных мобильных устройствах.

## Дорожная карта клиентов

| Клиент | Статус | Принятое решение |
|---|---|---|
| Web: десктоп и мобильный браузер | **Реализовано** | Next.js PWA использует общий `/v1` API и Socket.IO. |
| Telegram | **Будущее** | Telegram Bot для входа и приглашений плюс Telegram Mini App для игрового интерфейса. Telegram identity привязывается к существующему Dominoes account. |
| Android | **Будущее** | Приложение на Expo/React Native, использующее общий API, контракты и аккаунт. |
| iOS | **Будущее** | Приложение на Expo/React Native, использующее общий API, контракты и аккаунт. |

Нативные клиенты не должны реализовывать доверенную игровую логику самостоятельно. Сервер остаётся авторитетным для всех платформ.

## Правила текущей игры

Web MVP использует Draw Dominoes:

- набор double-six из 28 костей;
- два игрока, по семь костей каждому;
- добор до появления допустимого хода или окончания базара;
- партия состоит из раундов и продолжается до 100 очков;
- при выходе игрок получает сумму очков на костях соперника;
- при блокировке единственный игрок с меньшей суммой получает разницу сумм;
- при равенстве в заблокированном раунде никто не получает очков.

Полное нормативное описание, включая первый ход, пас, подсчёт очков и пограничные случаи: [GAME_RULES.md](GAME_RULES.md).

## Архитектурные решения

### Сервер — источник истины

- Клиент показывает состояние и отправляет намерение пользователя, но не утверждает результат хода.
- Сервер проверяет очередь, владение костью, допустимую сторону, версию состояния и права участника.
- Каждое принятое изменение увеличивает версию матча.
- Повторная команда с тем же idempotency key не должна применяться дважды.
- После reconnect клиент получает актуальный seat-specific snapshot и продолжает с его версии.

### Скрытая информация

- Игрок получает полный состав только своей руки.
- Для других мест передаётся только число оставшихся костей.
- Клиент видит размер базара, но не порядок и значения костей в нём.
- Секретные данные нельзя добавлять в логи, аналитику, ошибки или realtime payload.

### Аккаунты и идентичности

- `Account` — внутренняя учётная запись Dominoes.
- `AccountIdentity` связывает аккаунт со способом входа: Supabase сейчас, Telegram в будущем.
- `Profile` содержит уникальный публичный username и отображаемое имя.
- Username проверяется и резервируется сервером; клиентская проверка не гарантирует уникальность.
- `GuestSession` даёт ограниченный доступ только к приглашённому матчу.
- Гостевой участник в будущем сможет присоединить результат к полноценному аккаунту без создания второго Dominoes account.
- Смена способа входа не должна менять игровую историю и профиль.

### Приглашения

- Приватная ссылка содержит непрозрачный одноразовый токен.
- Сервер хранит безопасное представление токена и проверяет статус и срок действия.
- Использованное, отозванное или просроченное приглашение нельзя применить повторно.
- Приглашение не раскрывает руки, базар и внутренние идентификаторы участников.

### Данные

Prisma/PostgreSQL хранит аккаунты, идентичности, профили, гостевые сессии, матчи, места, команды, события, приглашения и результаты. Изменения схемы выполняются только миграциями. Ручное изменение production-схемы запрещено.

Команды и события дают аудит игрового процесса, а snapshot ускоряет восстановление. Транзакция должна согласованно сохранить команду, новое состояние и события.

## API и realtime-соглашения

- Публичные HTTP endpoints версионируются префиксом `/v1`.
- Входные и выходные payload валидируются общими Zod-схемами.
- Ошибки используют стабильный машинный `code`; локализованный текст формирует клиент.
- Несовместимое изменение требует новой версии контракта или миграционного периода.
- Клиент отправляет ожидаемую версию состояния и idempotency key для изменяющей команды.
- Stale version возвращает контролируемую ошибку и заставляет клиент обновить snapshot.
- Realtime-событие сообщает о новой версии; пропущенные данные восстанавливаются через API.
- Socket.IO не заменяет авторизацию: подключение и подписка проверяются сервером.

## Безопасность и приватность

- Секреты хранятся только в переменных окружения и secret stores платформ.
- Файлы `.env` и реальные ключи не коммитятся.
- Supabase JWT проверяется по подписи, issuer, audience и сроку действия.
- Guest principal имеет минимальные права и ограничен одним матчем.
- HTTP origins задаются явным allowlist через `WEB_ORIGINS`.
- Изменяющие операции имеют rate limits и correlation ID.
- Логи структурированы и не содержат токенов, содержимого скрытых рук и персональных данных без необходимости.
- Удаление аккаунта выполняется по [runbook](runbooks/account-deletion-ru.md).

## Локализация и доступность

- Поддерживаемые locale: `uk`, `en`, `ru`; locale по умолчанию — `uk`.
- Пользовательские строки не размещаются напрямую в компонентах, если они требуют перевода.
- Сервер возвращает коды, а не готовые локализованные сообщения.
- Новый пользовательский сценарий должен быть проверен во всех трёх locale.
- Цель доступности — WCAG 2.2 AA.
- Игровое поле должно поддерживать мышь, touch и клавиатуру.
- Состояние хода, выбранная кость, допустимые стороны и ошибки не должны передаваться только цветом.
- Интерактивные элементы получают доступные имена, видимый focus и достаточную область нажатия.

## Инфраструктура

### Реализовано в конфигурации репозитория

- Vercel для Next.js web;
- Render для долгоживущего NestJS API и текущей starter PostgreSQL-конфигурации в `render.yaml`;
- Supabase Auth для пользовательских идентичностей;
- Docker Compose для локального API и PostgreSQL;
- GitHub Actions для автоматических проверок;
- инфраструктурные файлы `vercel.json`, `render.yaml`, `Dockerfile` и `.github/workflows`.

Утверждённая целевая архитектура использует Supabase Pro и для production PostgreSQL. Перед production-релизом необходимо привести `DATABASE_URL` и инфраструктурные файлы к одному варианту и удалить лишнее определение Render database. Пока это не сделано, конфигурация Render PostgreSQL считается текущей реализацией, а перенос production-данных в Supabase — задачей статуса **Следующий этап**.

### Масштабирование

Первый production-релиз использует один экземпляр API. Redis добавляется только при переходе к нескольким экземплярам API, когда потребуется общий Socket.IO adapter, распределённые rate limits или координация фоновых задач.

## Инженерные соглашения

### Код

- TypeScript работает в строгом режиме; необоснованный `any` не используется.
- Имена в коде, API, схеме данных, комментариях и технических документах пишутся на английском.
- Пользовательская документация может быть русской или иметь отдельные локализованные версии.
- Общие правила и типы не копируются между приложениями, а выносятся в `packages`.
- Движок остаётся чистым и детерминированным: без сети, базы данных, времени системы и случайности без явного seed.
- Клиент не дублирует авторитетную проверку правил как источник истины.
- Новая зависимость добавляется только при ясной необходимости и фиксируется lock-файлом.

### Тестирование

- Изменение поведения начинается с теста, воспроизводящего требование или ошибку.
- Правила движка покрываются unit и property-based тестами.
- API-права, транзакции, idempotency, stale versions и скрытые данные покрываются integration tests.
- Критические пользовательские сценарии проверяются Playwright минимум в двух независимых browser contexts для матча людей.
- Исправление регрессии включает тест, который падал до исправления.
- Перед коммитом выполняется `pnpm check`; Prisma отдельно проверяется командой `pnpm --filter @dominoes/api db:validate` при изменении схемы.

### Git

- Основная ветка — `master`.
- Коммит должен быть небольшим, тематически цельным и не включать случайные пользовательские изменения.
- Сообщения используют понятный Conventional Commit-подобный префикс: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`.
- Секреты, собранные артефакты, временные файлы и локальные `.env` не коммитятся.
- Перед push проверяются `git diff`, `git status` и результат автоматических проверок.

### Документация

- Изменение правил обновляет `GAME_RULES.md` и тесты движка в том же коммите или pull request.
- Изменение архитектуры, стека, статуса функции или roadmap обновляет `PROJECT.md`.
- Изменение production-процедуры обновляет соответствующий runbook.
- Датированные specs и plans сохраняются как история и не подменяют актуальные документы.
- Документы не содержат неподтверждённых сроков и не выдают запланированную возможность за реализованную.

## Quality gates

Минимальная локальная проверка перед слиянием:

```powershell
pnpm check
```

Она включает lint, typecheck, unit/integration tests и production build. Дополнительно по области изменения выполняются:

```powershell
pnpm test:e2e
pnpm --filter @dominoes/api db:validate
k6 run tests/load/ai-match.js
```

E2E и load tests требуют подходящего окружения и запускаются отдельно от базовой проверки.
